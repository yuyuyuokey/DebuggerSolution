## 第1章 绪论

### 1.1 研究背景与意义

- **研究背景**：软件逆向工程在安全分析、漏洞挖掘与恶意代码检测中具有不可替代的作用。以 OllyDbg、x64dbg、IDA Pro 为代表的传统调试器功能强大但高度依赖使用者的汇编语义理解与逆向经验积累；分析师需同时具备指令级语义理解、算法模式识别与分支逻辑推断等多维能力，技术门槛高、分析效率低。
- **研究意义**：近年来大语言模型在代码语义理解方面展现出显著能力，但其在二进制逆向领域的应用仍以静态分析（反编译、符号恢复）和单次被动问答为主。本研究将大语言模型的代码语义理解与逻辑推理能力引入动态调试工作流，通过 Function Calling 机制将调试引擎底层能力封装为结构化工具集合，实现 AI 对调试器的自主调用，形成"AI 主动收集证据→推理分析→生成结论"的闭环，降低逆向分析的技术门槛，提升分析效率与自动化水平。

**建议引用**：[B2], [B3], [B4]

#### 1.2.6 现有工作的局限与本文切入点
1. 多数 AI + 逆向研究聚焦于**静态二进制分析**（反编译/符号恢复），AI 与**运行时调试器**的实时交互研究尚属空白
2. ChatDBG 是唯一与本文接近的工作，但其面向**源码级**调试；本文面向**二进制级**（无源码、无符号、仅汇编与内存）动态调试，技术难度更高
3. 现有 AI 辅助逆向多为**单次被动问答**，缺乏 AI 主动选择工具、收集证据、形成分析闭环的能力
4. 现有方案缺乏从"分析"到"自动修补"的**完整闭环**机制

### 1.3 主要研究内容

1. 设计并实现基于 Win32 Debug API 的用户态调试引擎，涵盖软件/硬件断点管理、四种粒度步进控制（步入/步过/执行到返回/执行到用户代码）、寄存器与内存读写、PE 结构解析、调用堆栈回溯、反反调试处理等完整动态分析功能
2. 构建基于 Function Calling 机制的大语言模型交互架构，将调试器底层能力封装为 17 个结构化工具（数据读取 8 个、语义分析 5 个、全局搜索 2 个、推理辅助 2 个），基于工具总线模式实现 AI 的自主工具选择与多轮闭环交互
3. 实现 AI 驱动的函数特征识别与评分排序、关键验证算法自动定位、分支路径语义加权分析、调用参数逆向恢复等智能化分析功能
4. 实现基于导入表扫描与字符串交叉引用的智能断点推荐机制，以及自然语言描述到形式化断点条件的自动转换
5. 设计并实现基于 MFC 多面板分割的图形化调试界面，完成 AI 分析结果的结构化展示、补丁确认交互与分析标注内联显示

## 第2章 相关理论与关键技术

### 2.1 基于 Win32 Debug API 的用户态调试机制

- Windows 调试子系统架构：内核态调试引擎（DbgKd）与用户态调试 API 的交互模型

- 进程创建调试（DEBUG_PROCESS）与进程附加调试（DebugActiveProcess）的机制差异与适用场景
- 引用 x64dbg 架构设计文档 [U14]

**建议引用**：[1], [2], [U1], [B3], [B4]

### 2.2 x86 架构下的断点实现原理

- INT3 软件断点：0xCC 单字节指令替换机制、异常捕获（EXCEPTION_BREAKPOINT, 0x80000003）与指令恢复策略、透明化处理流程（原始指令回写→EIP 回退→TF 单步→重新植入）
- 基于调试寄存器的硬件断点：DR0-DR3 地址寄存器（断点线性地址）、DR6 状态寄存器（断点命中标识位）、DR7 控制寄存器（L0-L3 局部启用位、G0-G3 全局启用位、RW0-RW3 读写条件位、LEN0-LEN3 长度配置位）
- 硬件断点三种触发条件：指令执行（00）、内存写入（01）、I/O 读写（10，仅限 CPL=0）、内存读写（11）
- 硬件断点长度配置：1 字节（00）、2 字节（01）、4 字节（11，部分架构支持 8 字节）
- 条件断点的表达式求值模型（寄存器比较、立即数匹配、组合逻辑）
- 临时断点的生命周期管理

**建议引用**：[10], [B3]

### 2.3 面向 x86 指令集的反汇编技术

- x86 变长指令编码格式：操作码映射（1-3 字节 opcode）、ModR/M 寻址字节（mod + reg/opcode + r/m）、SIB 比例变址字节（Scale + Index + Base）、位移量与立即数字段
- Zydis 反汇编引擎：Intel 语法输出、32 位传统模式解码（ZYDIS_MACHINE_MODE_LEGACY_32）、指令元数据提取（操作数可见性与访问类型）、指令长度计算

- 基于函数序言特征（PUSH EBP; MOV EBP, ESP）的函数边界自动识别算法
- LLM 辅助反混淆反汇编技术（DisasLLM [10] 的垃圾代码插入对抗策略）

### 2.4 PE 可执行文件格式解析

- 导出表（IMAGE_EXPORT_DIRECTORY）与函数名-地址三元组映射（Name/Ordinal/AddressOfFunctions）
- 原始入口点（OEP）的计算：ImageBase + AddressOfEntryPoint

## 第3章 基于LLM工具调用的动态分析框架设计

### 3.1 分层解耦的系统总体架构

- 三层分离式架构：
  - **调试引擎层**（DebuggerEngine.dll）—— 负责 Win32 调试事件循环、断点管理、反汇编缓存、进程/线程/内存操作
  - **AI 工具层**（AITools.dll）—— 负责工具注册与分发、Function Calling 交互循环、工具执行结果的 JSON 序列化
  - **图形界面层**（DebuggerGUI.exe）—— 负责多面板布局、反汇编可视化渲染、AI 分析结果展示与交互
- 各层间的数据流与调用关系：
  - GUI → 引擎：通过 C 导出函数（`dbg_*` API）进行调试操作
  - 引擎 → GUI：调试事件通过回调函数 + `PostMessage(WM_DEBUG_EVENT, ...)` 推送
  - GUI → AI 层：通过 `LoadLibrary` 动态加载 AITools.dll，调用 `InitAITools` / `ExecuteAITool`
  - AI 层 → 引擎：调用 `dbg_*` 导出函数收集调试上下文数据
  - AI 层 → DeepSeek API：通过 HTTPS POST（httplib.h）发送对话请求
- 层间通信机制对比：DLL 导出函数接口（同步）、函数指针回调（异步通知）、Windows 消息队列（跨线程）、JSON 数据交换格式（松耦合）
- 分层架构的设计原则：引擎层不依赖 GUI 实现（可独立复用）、AI 工具层可独立扩展（新增工具无需修改调用框架）、各层编译为独立二进制模块（物理解耦）

**建议引用**：[2], [30]

### 3.2 事件驱动的调试引擎模块设计

- 核心类职责划分：
  - `CMyDebug`：调试事件主循环（`WaitForDebugEvent` 阻塞等待→事件类型分发→`ContinueDebugEvent` 恢复执行）、进程创建/附加管理、PEB 反反调试、OEP 自动断点
  - `CBreakpointManager`：软件断点（INT3 0xCC 替换）与硬件断点（DR0-DR7 寄存器配置）的统一生命周期管理、ASLR 重定位、条件断点表达式存储
  - `CDisassembler`：Zydis 驱动的全量反汇编缓存、"地址→指令"二分检索、字符串引用智能探测注释、跨模块缓存切换
  - `CThreadManager`：目标进程中活跃线程的追踪（增/删）、硬件断点的全线程同步应用
- 基于哈希映射的断点管理模型：
  - 软件断点表：`unordered_map<地址, {原始字节, 启用状态, 条件表达式}>`
  - 硬件断点表：DR0-DR3 四槽位的分配与释放、DR7 位掩码的增量更新
  - 断点属性集：地址、类型（软件/硬件-执行/硬件-写入/硬件-访问）、条件表达式（可选）、启用状态
- 面向 ASLR 的断点地址重定位机制：
  - 基址偏移量 Δ = 当前 ImageBase - 上次会话 ImageBase
  - 所有软硬件断点地址统一施加 Δ 偏移
  - 进程创建事件（CREATE_PROCESS_DEBUG_EVENT）中立即执行重定位
- 基于 Zydis 的全量缓存反汇编模型：
  - .text 段一次性批量解码（ZydisDecoderDecodeFull）
  - 以指令地址为键构建有序数组（`vector<DecodedInstruction> + 二分查找`）
  - 跨模块切换时（如 CALL 到系统 DLL），切换对应模块的反汇编缓存
- 四种粒度的步进控制模型：
  - **步入**（StepInto）：TF 标志位置位，CPU 执行一条指令后触发 EXCEPTION_SINGLE_STEP
  - **步过**（StepOver）：反汇编缓存中检测当前指令是否为 CALL，若为 CALL 则在下一指令地址设置临时软件断点并全速运行
  - **执行到返回**（StepToReturn）：ESP 指向的返回地址 + EBP 链的双重验证 + 栈空间 512 条目扫描，在返回地址设临时断点
  - **执行到用户代码**（StepToUserCode）：栈 1024 条目深层扫描 + 模块边界检测，仅在主模块（.exe）地址范围内的返回地址设临时断点

### 3.3 基于工具总线模式的 AI 交互层设计

- 工具总线架构：工具注册表（`std::map<string, ToolHandler>` 名称→处理函数映射）、统一 JSON 参数接口（nlohmann/json）、标准化错误返回格式（`{"error": "..."}`）
- 17 个工具的功能划分与设计意图：

| 类别 | 工具名 | 功能描述 | 设计意图 |
|------|--------|---------|---------|
| **数据读取** (8) | `read_assembly` | 反汇编指定地址 5-120 行指令 | 提供代码层面的基本观察能力 |
| | `read_registers` | 读取全部 10 个 x86 通用寄存器及 EFLAGS | 提供执行状态的即时快照 |
| | `read_memory_hex` | 以十六进制 Dump 格式读取 1-256 字节内存 | 提供数据层面的观察能力 |

## 第4章 系统关键模块实现

### 4.1 调试引擎核心机制的实现

#### 4.1.1 调试事件循环的主控流程
- `WaitForDebugEvent(&DebugEvent, INFINITE)` 阻塞等待
- `switch (DebugEvent.dwDebugEventCode)` 八种事件类型分发
- 各事件处理函数的职责与返回值（`DBG_CONTINUE` / `DBG_EXCEPTION_NOT_HANDLED`）
- `ContinueDebugEvent(dwProcessId, dwThreadId, dwContinueStatus)` 恢复执行
- 事件循环的线程模型：后台工作线程运行事件循环，通过 `PostMessage` 将暂停事件通知 GUI 主线程

#### 4.1.2 INT3 软件断点的透明化实现
- **设置**：`ReadProcessMemory` 保存原始字节 → `WriteProcessMemory` 写入 0xCC
- **命中处理**：EXCEPTION_BREAKPOINT (0x80000003) 捕获 → 查询断点表确认是否为本调试器设置的断点
- **透明恢复**：原始指令回写（恢复被覆盖的字节）→ EIP 单字节回退（0xCC 是一字节指令，回退后 CPU 从原地址重新执行）→ EFLAGS.TF 置位（第 8 位，触发单步执行）→ `ContinueDebugEvent` 恢复运行 → 在紧随的 EXCEPTION_SINGLE_STEP 中重新植入 0xCC
- **实现要点**：内存保护属性可能为 PAGE_EXECUTE_READ，需临时调整为 PAGE_EXECUTE_READWRITE；多线程场景下，断点命中时其他线程可能也执行到同一地址，需要正确的异常过滤

#### 4.1.3 基于调试寄存器的硬件断点实现
- DR0-DR3 四槽位的分配策略：首次设置时使用 DR0，逐次填充
- DR7 控制寄存器的位级配置：
  - L0-L3（位 0/2/4/6）：局部启用，任务切换时自动清除
  - G0-G3（位 1/3/5/7）：全局启用，任务切换时保留
  - RW0-RW3（位 16-17/20-21/24-25/28-29）：00=执行, 01=写入, 11=读写
  - LEN0-LEN3（位 18-19/22-23/26-27/30-31）：00=1字节, 01=2字节, 11=4字节
- 全线程同步：通过 `CThreadManager` 遍历当前所有线程，对每个线程调用 `SetThreadContext(CONTEXT_DEBUG_REGISTERS)` 设置相同的 DR0-DR7
- 命中判定：EXCEPTION_SINGLE_STEP (0x80000004) 触发后，逐线程读取 DR6 寄存器的 B0-B3 位，确定具体哪个断点被命中
- 硬件断点触发的特殊性：对于写入/访问断点，异常在写入/访问指令**执行后**触发（指令已生效），EIP 指向下一条指令

#### 4.1.4 四种粒度步进控制的实现
- **步入（StepInto）**：设置 EFLAGS.TF = 1（置位第 8 位）→ `ContinueDebugEvent` → CPU 执行一条指令后自动触发 EXCEPTION_SINGLE_STEP
- **步过（StepOver）**：在反汇编缓存中查找当前 EIP 对应的指令 → 判断操作码是否为 CALL（0xE8 直接调用 / 0xFF 间接调用）→ 若为 CALL，在返回地址（当前指令地址 + 指令长度）设置临时软件断点 → 全速运行 → 断点触发后自动清除
- **执行到返回（RunToReturn）**：三重验证机制：
  1. 从 ESP 读取栈顶的返回地址
  2. 在反汇编缓存中验证返回地址前是否存在 CALL 指令（回退 5-7 字节搜索 0xE8 或 0xFF 0x15 模式）
  3. EBP 链验证（当前 EBP 指向的栈帧中保存的返回地址是否一致）
  4. 栈 512 条目扫描以防返回地址不在栈顶
- **执行到用户代码（RunToUserCode）**：栈 1024 条目深度遍历 → 遍历每个可能的返回地址 → 检查地址是否在主模块（.exe）范围内 → 在第一个符合条件的地址设临时断点

#### 4.1.5 反汇编缓存与检索实现
- 批量解码：以 .text 段起始地址为起点，遍历整个代码段，逐条调用 `ZydisDecoderDecodeFull` 解码
- 缓存数据结构：`vector<pair<DWORD, ZydisDecodedInstruction>>`，按地址递增存储
- 二分检索：`lower_bound` 实现 O(log n) 的指令地址查找
- 指令注释增强：识别 LEA/MOV/PUSH 等指令中的立即数是否落在字符串交叉引用的已知地址范围内，若是则在注释中附加字符串内容
- 跨模块缓存：`unordered_map<HMODULE, DisassemblyCache>`，DLL 加载时自动为该模块构建缓存

#### 4.1.6 PEB 反反调试处理
- PEB 获取路径：`NtQueryInformationProcess(hProcess, ProcessBasicInformation, &pbi, ...)` → `pbi.PebBaseAddress`
- 关键标志清零：
  - `BeingDebugged`（PEB 偏移 0x02）：置 0，绕过 `IsDebuggerPresent()` API
  - `NtGlobalFlag`（PEB 偏移 0x68）：置 0，绕过堆标记检测（正常进程为 0，被调试进程为 0x70）
- 执行时机：`CREATE_PROCESS_DEBUG_EVENT` 事件中，在目标进程任何代码执行前完成

#### 4.1.7 OEP 自动断点与 ASLR 断点重定位
- OEP 自动断点：解析 PE 头 → 读取 IMAGE_OPTIONAL_HEADER32.AddressOfEntryPoint → 计算 OEP = ImageBase + AddressOfEntryPoint → 在 OEP 设置临时软件断点
- ASLR 重定位：记录本次 ImageBase，计算 Δ = 本次 ImageBase - 上次存储的 ImageBase → 遍历所有已注册的软硬件断点 → 对每个断点地址施加 Δ → 更新断点哈希表

### 4.2 AI 工具调用闭环机制的实现

#### 4.2.1 工具总线架构
- AITools.dll 的三个导出函数：`InitAITools(hProcess, dwPID)` 初始化执行上下文、`ExecuteAITool(toolName, jsonArgs)` 工具分发执行、`SetCurrentThreadId(dwThreadId)` 更新当前线程 ID
- 工具注册机制：`static std::map<std::string, ToolHandler>` 静态全局注册表，每个 `ToolHandler` 为 `std::function<std::string(const nlohmann::json&)>`
- JSON 参数解析与类型校验：`json.at("address").get<uint32_t>()` 带异常捕获
- 异常边界统一捕获：工具函数内部 try-catch，异常时返回 `{"error": "message"}` JSON

#### 4.2.2 核心分析工具的算法实现

**read_validation_checkpoints —— CMP/TEST + Jcc 指令对扫描算法**
- 遍历函数反汇编指令数组
- 识别模式：`CMP reg/mem, imm/reg` 或 `TEST reg, reg` → 紧随的条件跳转指令（JE/JNE/JG/JL/JAE/JBE 等 16 种 Jcc）
- 记录检查点信息：比较指令地址、跳转指令地址、跳转目标地址、条件类型
- 上限控制：最多记录 16 个检查点（防止大函数返回过于冗余的信息）

**read_best_algorithm_callee —— 多维特征加权评分算法**
- 针对每个候选子函数计算加权总分：
  - 特征标签匹配：验证标记 +30、XOR 循环 +35、CRC32 +28、Base64 +24、加密 +25
  - 比较指令密度：最多 6 条，每条 +4 分
  - 条件跳转密度：最多 6 条，每条 +3 分
  - 循环结构出现：最多 3 个，每个 +7 分
  - 算术运算指令数：最多 10 条，每条 +3 分
  - 验证关键词字符串引用：每匹配一个 +12 分
- 惩罚项：CALL 指令超过 10 条开始扣分（防止将简单调度函数误判为核心算法）、指令数少于 5 条扣分（过滤桩函数）
- 排序后返回 Top 3 的摘要信息

**get_call_parameters —— PUSH 指令反向扫描参数恢复**
- 从 CALL 指令地址向前（低地址方向）扫描
- 收集 PUSH 指令的立即数或寄存器 → 按 PUSH 顺序对应参数（从右到左的 cdecl/stdcall 调用约定）
- 正向辅助：从函数返回后的 ADD ESP, N 推断总参数字节数 → 除以 4 得参数个数
- 输出参数列表：（序号，PUSH 地址，值类型[立即数/寄存器/内存]，值内容）

**分支路径评分机制（read_validation_paths）**
- 对于每个验证检查点，追踪两个分支的前 3 条指令
- 检测分支路径上引用的字符串，使用语义字典进行加权：
  - 成功暗示词（SUCCESS/GOOD/CORRECT/REGISTERED/OK）：+10 分
  - 失败暗示词（FAIL/WRONG/INVALID/BAD/ERROR/INCORRECT）：-10 分
  - 中性词（PASSWORD/SERIAL/LICENSE/NAME）：+3 分
- 综合评分推断：条件成立分支为正分 → 该条件对应"验证通过"；条件不成立分支为正分 → 该条件对应"验证通过"

#### 4.2.3 Function Calling 多轮交互引擎

- HTTP POST 请求构造（httplib.h）：
  ```json
  {
    "model": "deepseek-coder",
    "messages": [{"role": "system", "content": "..."}, ...],
    "tools": [{ "type": "function", "function": { "name": "...", "description": "...", "parameters": {...} } }, ...],
  ```

- 响应解析：判断 `finish_reason == "tool_calls"` → 提取 `tool_calls[]` 数组 → 解析每个 `function.name` 和 `function.arguments`
- Tool Call 结果拼装：以 `role: "tool"` 消息格式将工具执行结果追加到对话历史
- 轮次阈值控制：硬上限 12 轮（防止无限循环和 Token 超支）
- 证据完整性判定：当 AI 已收集当前函数的摘要（read_function_summary）、验证检查点（read_validation_checkpoints）、最优候选函数的深度分析（read_best_algorithm_callee + 对应的验证路径分析），在下一次 API 请求中通过 System Prompt 追加"请基于已有证据输出最终分析结论和 Patch 方案"的引导
- 后台线程异步执行：`std::thread` detach 模式，AI 分析不阻塞 GUI 主线程

#### 4.2.4 AI 驱动内存补丁的自动应用

- Patch 提取：正则匹配回复文本中的 `{"patch_addr": "0x...", "hex_bytes": "..."}` JSON 模式
- 补丁合法性校验：目标地址是否在主模块地址空间内、内存区域属性是否为可执行（PAGE_EXECUTE_READ 等）
- 安全写入流程：`VirtualProtectEx` 修改页面保护（添加 PAGE_EXECUTE_READWRITE）→ `WriteProcessMemory` 写入补丁字节 → `VirtualProtectEx` 恢复原始页面保护
- 用户确认对话框：展示补丁地址、原始字节（十六进制）、补丁字节（十六进制），用户确认后执行写入

### 4.3 关键问题的分析与解决

#### 4.3.1 进程重启导致的断点地址失效 → ASLR 增量重定位
- **问题**：每次以 DEBUG_PROCESS 模式重新启动目标程序时，ASLR 导致 ImageBase 随机化，前次调试会话中的所有断点地址失效
- **解决方案**：在进程创建事件中记录当前 ImageBase，与上次存储的 ImageBase 比较计算 Δ，遍历所有软硬件断点地址施加统一的 Δ 偏移
- **实现细节**：`unordered_map<DWORD, BreakpointInfo>` 中的键（地址）全部更新；硬件断点的 DR0-DR3 重新设置

#### 4.3.2 AI 对系统模块汇编的无意义分析 → 地址空间边界防护
- **问题**：AI 可能对 kernel32.dll、ntdll.dll 等系统 DLL 中的汇编代码进行分析，既无逆向价值也浪费 API Token
- **解决方案**：在工具层设置地址上限过滤（0x50000000），所有工具的地址参数先通过边界检查；系统 DLL 通常加载在 0x70000000 以上的高地址空间
- **实现细节**：`AIToolsContext` 中存储主模块的加载基址与大小，工具函数调用前检查目标地址是否在主模块范围或 0x50000000 以下

#### 4.3.3 单次 AI 调用缺乏全局分析上下文 → 多级证据预取策略
- **问题**：如果在 AI 第一轮对话时仅提供当前 EIP 和寄存器值，AI 缺乏足够的信息来判断"应该先调用哪个工具"
- **解决方案**：在发起 Function Calling 请求前，自动调用以下 API 收集预处理信息：
  1. 当前函数的反汇编摘要（指令数、特征标签）
  2. 所有被调用子函数的完整列表
  3. 对前 5 个子函数运行最优候选评分算法，预分析 Top 3
  4. 为 Top 3 候选函数预先计算验证检查点和路径评分
- 这些预取证数据以"你已经预先收集了以下信息"的形式注入 System Prompt

#### 4.3.4 临时断点在非目标模块的误触发 → EIP 范围与模块归属校验
- **问题**：RunToReturn/RunToUserCode 在执行过程中，可能因为异常/中断导致执行流转到非预期的系统 DLL 区域，临时断点在这些位置触发会混淆用户
- **解决方案**：
  - 临时断点命中时，验证当前 EIP 是否在预期的地址窗口内（如返回地址 ±16 字节缓冲区）
  - RunToUserCode 额外要求 EIP 在主模块（.exe）范围内
  - 若校验失败，清除匹配但未落入范围的临时断点，继续执行

### 4.4 智能化辅助分析功能的实现

#### 4.4.1 AI 智能断点推荐
- 导入表分析：解析目标程序的 PE 导入表 → 筛选敏感 API 类别（文件操作：CreateFileA/ReadFile/WriteFile、注册表：RegOpenKeyExA/RegQueryValueExA、进程：WinExec/CreateProcessA、网络：WSASend/send/recv、字符串：lstrcmpA/strcmp/wcscmp）→ 自动推荐在这些 API 入口处设置断点
- 字符串交叉引用扫描：在反汇编缓存中搜索 100+ 安全性关键词（password/serial/key/license/注册/密码/序列号/trial/activate/login/verify/check/auth）+ 逆向语义关键词（wrong/invalid/error/fail/success/correct/valid）→ 列出引用这些字符串的指令地址作为推荐断点
- 用户界面：CAIBreakpointRecommendDlg 以复选框列表展示推荐断点，支持全选/取消，一键批量设置

#### 4.4.2 自然语言条件断点生成
- 用户输入示例："当 EAX 等于字符串 'Admin' 的地址时断下" / "在 ECX 大于 0x1000 时暂停" / "当 ESI 指向 'Password' 字符串时断下"
- AI 解析流程：将用户自然语言描述发送给 DeepSeek → System Prompt 引导模型输出结构化的断点信息：`{"breakpoint_type": "hardware_execute", "address": "0x401000", "condition": "eax == 0x405000"}` → 系统解析 JSON → 设置对应的条件断点
- 实现：CNaturalLanguageBPDlg 输入框 + 自定义 System Prompt（引导模型解析用户意图为结构化断点参数）

#### 4.4.3 AI 函数特征识别与标注
- 函数边界识别：基于 PUSH EBP; MOV EBP, ESP 序言模式 + Zydis 解码遍历
- 函数特征分类：通过 `read_function_summary` 工具自动检测 XorLoop（XOR 循环加密）、StrCmp（字符串比较链）、Base64（Base64 编码表引用）、CRC32（CRC 查表算法）、MemCopy（批量内存操作）等特征
- 在反汇编视图中的内联显示：CAIFunctionListDlg 展示函数名、起止地址、指令数、置信度，支持双击跳转到函数入口

## 第5章 实验测试与效果评估

### 5.1 测试环境与评价指标

- **硬件平台**：处理器型号、内存容量、操作系统版本（Windows 11/10, x64 宿主运行 x86 目标）
- **软件栈**：Visual Studio 2022 编译环境、Windows SDK 10.0、DeepSeek API（deepseek-coder 模型）、Zydis v4.x
- **测试目标程序选取**：3 个不同难度梯度的 x86 CrackMe 程序
  - CM1：简单字符串对比型（直接 strcmp，序列号硬编码在数据段）
  - CRACKME：XOR 加密算法型（序列号经过 XOR 变换后与预设值比较）
  - CRACKME_PACKED / abexcm5：多阶段验证 + 反调试保护型（含 PEB BeingDebugged 检测 + 多级跳转混淆）
- **多维评价指标体系**：
  - 基础功能正确性：断点命中率、步进精度、寄存器/Memory 读写一致性
  - AI 分析准确率：验证算法识别正确率、关键跳转定位正确率、Patch 有效性
  - 分析任务耗时：从触发分析到输出结论的总时间（含 API 调用延迟）
  - 工具调用轮次分布：完成分析所需的 Function Calling 轮次
  - 人工对比基线：有经验逆向人员的操作步骤数与总耗时

### 5.2 核心调试功能的正确性验证

- **进程创建调试与附加调试的启动链路测试**：分别以 DEBUG_PROCESS 模式启动 CM1 和以 DebugActiveProcess 附加到运行中的 CRACKME，验证两种模式均能正常拦截 OEP 并进入暂停状态
- **软件断点完整生命周期测试**：在目标地址设置 INT3 断点 → 全速运行 → 验证 EIP 准确停在断点地址 → 单步执行 → 验证原始指令正确执行 → 验证断点被重新植入
- **硬件断点三种触发模式测试**：
  - 执行断点：在指定地址设置硬件执行断点 → 运行 → 验证在目标地址暂停
  - 写入断点：在数据变量地址设置硬件写入断点 → 运行 → 验证在修改该变量的指令处暂停
  - 访问断点：在敏感数据区域设置硬件访问断点 → 运行 → 验证在读或写该区域的指令处暂停
- **四种步进粒度的控制精度测试**：
  - 步入：执行 CALL 指令时验证进入子函数内部
  - 步过：执行 CALL 指令时验证跳过子函数，停在 CALL 的下一指令
  - 执行到返回：在子函数内部执行 → 验证停在调用该子函数的 CALL 指令的下一条
  - 执行到用户代码：在系统 DLL 函数中执行 → 验证回到主模块代码空间
- **寄存器读写一致性测试**：通过直接修改窗口修改 EAX=0x12345678 → 读取寄存器 → 验证返回 0x12345678 → 在目标进程中验证 GetThreadContext 返回一致的值
- **内存布局枚举完整性测试**：通过 VirtualQueryEx 枚举内存映射 → 与 Process Explorer 显示结果对比
- **调用堆栈回溯的层数准确性测试**：在已知深度的函数调用链中暂停 → 验证堆栈回溯的帧数与函数名（通过 EBP 链遍历）
- **PEB 反反调试有效性测试**：在 CRACKME_PACKED（含 PEB BeingDebugged 检测）上启动调试 → 验证程序不会因检测到调试器而退出

## 第6章 总结与展望

### 6.1 工作总结

1. 设计并实现了基于 Win32 Debug API 的用户态调试引擎，具备 INT3 软件断点与 DR0-DR3 硬件断点的完整管理、四种粒度步进控制（步入/步过/执行到返回/执行到用户代码）、寄存器与内存读写、PE 结构解析与导入表枚举、EBP 链调用堆栈回溯、PEB 反反调试处理、ASLR 断点自适应重定位等完整的动态分析功能
2. 构建了包含 17 个结构化工具的 AI 交互层（数据读取 8 个、语义分析 5 个、全局搜索 2 个、推理辅助 2 个），基于工具总线架构和 Function Calling 机制实现了大语言模型与调试引擎的多轮自主闭环交互，包括上下文预取、证据完整性判定和安全边界防护
3. 实现了 AI 驱动的函数特征识别与多维评分排序、验证检查点自动定位与分支路径语义分析、调用参数逆向恢复、字符串解密模式检测等智能化分析功能，以及从 AI 分析结论到内存补丁自动应用的完整闭环
4. 实现了基于导入表敏感 API 扫描与字符串交叉引用的智能断点推荐机制，以及自然语言描述到形式化断点条件的自动转换
5. 基于 MFC 框架设计了多面板分割的图形化调试界面，实现了反汇编语法语义联合高亮、AI 标注内联显示、分析结果结构化展示与交互确认等功能

### 6.2 不足与展望

- **架构支持范围**：当前仅支持 32 位 x86 目标程序，未来可扩展至 x64 架构（AMD64）以覆盖现代应用场景，需要适配 x64 调用约定（RCX/RDX/R8/R9 传参）、64 位地址空间和 WoW64 调试子系统
- **模型部署方式**：AI 分析能力依赖云端 DeepSeek API 调用，存在网络延迟（单次 API 往返 2-5 秒）和二进制代码隐私风险，未来可探索本地模型部署方案。REx86 [14] 已证明 Qwen2.5-Coder-7B 通过领域微调可在 x86 汇编辅助分析任务上取得可用效果，ReCopilot [13] 也展示了 7B 级多任务二进制分析模型的潜力
- **反反调试深度**：当前反反调试覆盖 PEB 层面的 BeingDebugged 和 NtGlobalFlag 清零，未来可扩展至更全面的反检测体系（如 NtSetInformationThread 隐藏线程、NtQuerySystemInformation 隐藏进程、硬件断点扫描检测对抗、时间戳反调试绕过等），参考 ScyllaHide 的多策略协同方案
- **数据流分析精度**：当前数据流追踪（trace_data_flow）采用线性指令扫描方式，受限于基本块内的定义-使用分析，未来可引入中间表示（IR）级别的数据流分析或符号执行引擎（如 angr/Triton）以提升跨基本块的数据流追踪精度，参考 LLM vs 符号执行 [U5] 的互补策略
- **交互模式升级**：当前自然语言交互为单次请求-响应模式，未来可扩展为多轮会话式调试助手（类似 CAIResultDlg 的扩展），实现上下文中持续追问、逐步深入的分析体验
- **Agent 化演进**：参考 ClearAgent [39] 的 Agentic 二进制分析和 FIRMHIVE [40] 的递归 Agent 分析，未来可引入多 Agent 协作架构（分析 Agent + 验证 Agent + 反馈 Agent），实现更复杂的自主逆向分析策略
